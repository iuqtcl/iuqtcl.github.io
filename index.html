  • Done() 用于完成 mapreduce 后停止 coordinator

查询执行

火山/迭代模型 Volcano Model

是解释计算模型，将关系代数中的每一种操作抽象为一个 Operator ，将整个 SQL 构建成一个 Operator 树，从根结点到叶子结点自上而下递归调用 next() 。

与物化模型相对，物化模型将每一步物理操作实体化（全部完成）后，用于下一步输入。

  • 火山模型将物理操作符的结果元组直接发送给下一个操作符使用。
  • 每一次传递一批元组，是向量化模型。（攒够一批传递），更好地利用现代 CPU 的 SIMD 指令。

火山模型样例

每个Operateor调用子结点，获得子结点的返回值之后进行操作，返回给上游。

火山模型优缺点

优点： 每个 Operator 可以单独抽象实现，不需要关心其他 Operator 的逻辑。

缺点： Tuple-at-a-time ,多次调用 next() ，造成大量的虚函数调用。

1.2 语义、解释器和栈式虚拟机

堆、栈 内存划分

栈内存

每个函数被调用时，创建一个对应本次函数调用的 stack frame 并 push 到栈内存上，当这个函数返回时 pop。

每个栈帧的大小等于当前函数所有的形参和局部变量大小的总和加元数据。变量名在编译时转换为偏移量。

多线程程序，每个线程都有自己的栈内存。

优势:

  1. 无需通过指针或引用动态寻址，访问速度快。
  2. 生命周期明确，内存空间使用效率高。
  3. 大小和偏移量固定。
  4. 不需要复杂的垃圾回收机制。

特性：

  1. 函数不能返回一个指向栈内存的结构体的指针。因为返回后数据就被删除了。（但对于C/C++，形参或局部变量是一个指针，变量大小只考虑指针本身的大小，不考虑指针指向的数据的大小）
  2. 无法在多个线程间共享。
  3. 大小较小（几MB），函数调用过深或结构体过大会 stack overflow

堆内存

和数据结构中的堆没有关系。

按需分配的动态内存管理机制。申请时再分配（相对于栈内存调用函数时一次性分配）。因此访问时需要通过指针或引用。

常见堆内存管理手段

  1. 程序员手动管理（C/C++)
  2. 垃圾回收机制
  3. 程序员明确指出每个变量的声明周期，并引入结构体的转让/借用机制。（Rust）

什么是(高级语言)虚拟机（HLL VM)

用于实现高级语言的语义。（主流HLL VM，诸如JVM、CLI、CPython、CRuby 1.9）

输入： 满足某种指令集架构的指令序列 （可能由编译器生成）

处理： 转换为目标ISA的指令序列并加以执行（编译/解释实现）
